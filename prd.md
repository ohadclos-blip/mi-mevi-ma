# PRD: מי מביא מה (BringIt)
## מסמך דרישות מוצר — גרסה 1.1

---

## תקציר מנהלים

### הבעיה
ארגון אירועים קהילתיים משותפים (מסיבות כיתה, פיקניקים, ארוחות שיתופיות) כרוך בתיאום כבד. המארגנים מפרסמים רשימות בצ'אט (לרוב וואטסאפ), מה שמוביל לכאוס: כפילויות, פריטים שנשכחו ומעקב ידני מתיש.

### הפתרון
אפליקציית ווב קלת-משקל ומותאמת למובייל לפישוט תכנון אירועים משותפים. עיקרון המנחה: **אפס חיכוך (Zero Friction)**.

- **לאורחים:** ללא הרשמה, סיסמה, או הורדת אפליקציה. לוחצים על קישור, מכניסים שם, ובוחרים פריט — תוך שניות.
- **למארגנים / משתמשים מתקדמים:** הרשמה רשותית לצורך ניהול אירועים מרובים ומעקב מצטבר.

---

## משתמשים ופרסונות

### פרסונה 1: המארגן (Organizer)
- מארגן אירוע בית-ספרי, קהילתי, או משפחתי
- **חייב להיות מחובר עם Google** — יצירת אירוע מחייבת הרשמה
- צריך לייצר רשימה, לשתף, ולעקוב אחרי מה שאנשים התחייבו להביא
- מכיר ווב בסיסי; מפעיל בעיקר ממובייל

### פרסונה 2: האורח האנונימי (Anonymous Guest)
- מקבל קישור בוואטסאפ
- פותח דפדפן ממובייל, מכניס כינוי, בוחר/מוסיף פריט
- לא רוצה להירשם — כל חיכוך מוביל לנטישה

### פרסונה 3: האורח הרשום (Returning Registered Guest)
- השתמש בעבר כאורח אנונימי ובחר לשמור היסטוריה
- מחובר עם Google, רואה לוח בקרה אישי
- יכול גם ליצור אירועים בעצמו

## זרימות משתמש מרכזיות

### זרימה 1: יצירת אירוע (המארגן)
> **הרשמה עם Google היא תנאי מוקדם ליצירת אירוע.** משתמש לא מחובר המנסה ליצור אירוע יועבר אוטומטית לזרימת ה-OAuth לפני הכניסה לטופס.
```
התחברות (Google Auth) — חובה
    → לחיצה על "אירוע חדש"
    → הזנת שם אירוע, תאריך, תיאור קצר
    → בחירת מצב רשימה (מוגדרת מראש / פתוחה)
    → [אם מוגדרת מראש] הזנת פריטים וכמויות
    → קבלת URL ייחודי
    → שיתוף דרך כפתור "שתף לוואטסאפ"
```

### זרימה 2: הצטרפות כאורח אנונימי
```
לחיצה על URL מוואטסאפ
    → מסך פשוט: "מה שמך?" (כינוי)
    → [אם LocalStorage זמין] טעינה אוטומטית של שם קודם
    → צפייה ברשימת הפריטים
    → בחירת פריט פנוי / הוספת פריט חדש
    → אישור → סיום
```

### זרימה 3: שדרוג רשות לחשבון
```
אורח אנונימי מסיים בחירה
    → הצעה: "רוצה לשמור את ההיסטוריה שלך?"
    → לחיצה על "כן, התחבר עם Google"
    → OAuth
    → מיזוג LocalStorage לחשבון ענן
    → הפניה ל"לוח הבקרה שלי"
```

## מודולים פונקציונליים

### מודול A: ניהול אירועים ורשימות

#### מצב 1: רשימה מוגדרת מראש (Top-Down)
- המארגן מגדיר פריטים וכמויות (לדוגמה: "בורקס — 2 מגשים", "קולה — 4 בקבוקים")
- אורחים בוחרים פריט פנוי ומשייכים אותו לשמם
- פריטים שנלקחו מסומנים ויזואלית כ"תפוסים"
- אפשרות לחלוקת פריט בין כמה אנשים (לפי כמות)

#### מצב 2: רשימה פתוחה (Bottom-Up)
- בד ריק — האורחים מוסיפים בעצמם מה הם מתכוונים להביא
- מניעת כפילויות מדויקות: התראה אם פריט זהה כבר קיים ברשימה
- המארגן יכול לערוך/למחוק פריטים

#### שמירת עקביות (Concurrency Guard)
- נעילה אופטימיסטית: שני משתמשים לא יכולים לתפוס אותו פריט בו-זמנית
- במקרה של קונפליקט — המשתמש השני מקבל הודעה ומתבקש לבחור מחדש

#### ביטול בחירה (Unclaim)
- כל אירוע כולל הגדרה בינארית של המארגן: **"אפשר לבטל בחירה"** / **"לא ניתן לבטל"**
- כשמותר: האורח רואה כפתור "בטל" לצד הפריט שבחר
- כשאסור: הפריט מוצג כ"תפוס סופית" ואין אפשרות ביטול עצמאי
- המארגן תמיד יכול לבטל בחירה של כל אורח, ללא תלות בהגדרה

#### מחזור חיי אירוע (Event Lifecycle)
אחרי שתאריך האירוע עובר, המערכת שולחת התראה למארגן עם 3 אפשרויות פעולה:

| פעולה | תיאור | מה האורח רואה |
|-------|--------|----------------|
| **מחיקה** | האירוע נמחק לחלוטין | שגיאת 404 / "אירוע לא נמצא" |
| **הקפאה** | האירוע קריא בלבד — לא ניתן לשנות | רשימה סגורה, סימון "הסתיים" |
| **חסימה** | הקישור חסום, אף אחד לא יכול לגשת | הודעת "גישה חסומה" |

- ברירת המחדל: האירוע נשאר פעיל עד פעולה מפורשת של המארגן (אין מחיקה אוטומטית)
- אירוע מוקפא שומר את כל הנתונים ומאפשר **שכפול לאירוע הבא**

#### נראות בחירות (Visibility Settings)
- כל אירוע כולל הגדרת נראות שהמארגן מגדיר בעת יצירת האירוע (וניתן לשנות לאחר מכן):
  - **מצב פתוח (ברירת מחדל):** כל המשתמשים — אורחים ומארגן — רואים את מלוא הרשימה כולל שמות הבוחרים ("דני — עוגת שוקולד", "מירה — מיץ ×2")
  - **מצב פרטי:** כל משתמש רואה רק את הבחירה שלו עצמו; שאר הפריטים מוצגים כ"תפוס" ללא זיהוי. המארגן רואה הכל תמיד.
- שינוי הגדרה לאחר שאורחים כבר בחרו — מוציא/מכניס תצוגת שמות בזמן אמת, ללא שינוי הנתונים

#### שכפול רשימה לאירוע חדש (Clone Event)
- מתוך אירוע קיים (פעיל, מוקפא, או כל אירוע שבניהול המארגן) — כפתור "שכפל לאירוע חדש"
- הפעולה יוצרת אירוע חדש עם כל הפריטים מהרשימה המקורית, כולל הכמויות, **ללא** הבחירות (הכל פנוי מחדש)
- המארגן יכול לערוך את הרשימה לפני פרסום האירוע החדש
- כל הפריטים מועתקים — כולל כאלה שאורחים הוסיפו ב-open list
- כל הבחירות מתאפסות — הרשימה החדשה מתחילה ריקה לחלוטין מבחינת שיוך
- שמות האורחים מהאירוע הישן לא מועתקים

### מודול B: גישה ללא חיכוך

#### URL ייחודי
- כל אירוע מקבל UUID4: `/event/d3b07384-d113-4ec6-a5d7`
- הURL הוא גם מזהה האירוע וגם "טוקן" הגישה
- אין צורך בסיסמה נוספת לצפייה ולהשתתפות

#### זיהוי מקומי (LocalStorage)
- שם הכינוי של האורח נשמר ב-LocalStorage
- בביקור חוזר (לאותו אירוע או לאירוע אחר) — הכינוי נטען אוטומטית
- שמירה גם של פריטים שהאורח בחר, לכל אירוע בנפרד

### מודול C: לוח הבקרה האישי (My Board)

גלוי רק למשתמשים רשומים. מחולק לשניים:

#### "אירועים שאני מנהל"
- רשימת כל האירועים שנוצרו על ידי המשתמש
- לכל אירוע: מספר משתתפים, % כיסוי הרשימה, כפתור עריכה
- גישה מהירה לקישור השיתוף

#### "אירועים שאני משתתף"
- רשימת כל האירועים שהצטרף אליהם (כאורח רשום)
- לכל אירוע: תאריך, שם, סיכום מה התחייב להביא
- דוגמה: "מסיבת סוף שנה — מביא: עוגת שוקולד ×1"

#### מיזוג היסטוריה (Data Migration)
- בהתחברות ראשונה, קריאת LocalStorage ומיזוג האירועים האנונימיים לחשבון הענן
- אירועים אנונימיים מקושרים לחשבון רק אם ה-UUID של האירוע עדיין תקף

### מודול E: דשבורד מנהל מערכת (Admin Dashboard)

פאנל פנימי חשוף **אך ורק לאדמין** (משתמש יחיד מוגדר ב-config לפי UID של Google). נגיש דרך נתיב מוסתר, לדוגמה `/admin`, עם בדיקת הרשאה קשיחה בשרת.

#### נתוני אירועים
- טבלת כל האירועים במערכת: שם, תאריך, מצב (active/frozen/blocked/deleted), מארגן, מספר משתתפים, % כיסוי רשימה
- פילטור לפי מצב / תאריך / מארגן
- צפייה בפירוט פריטים ובחירות של כל אירוע

#### נתוני משתמשים רשומים
- טבלת כל המשתמשים שנרשמו עם Google: שם, אימייל, תאריך הרשמה, מספר אירועים שיצרו, מספר אירועים שהשתתפו בהם

#### נתוני פריטים (אגרגטיביים)
- פריטים נפוצים ברחבי המערכת (לצורך ניתוח עתידי, לדוגמה — אוטו-קומפליט חכם)
- ספירת כמה פעמים כל פריט נבחר בסך הכל

#### כלי ניהול
- יכולת שינוי מצב אירוע ידנית (הקפאה / חסימה / מחיקה) — לצורך מודרציה
- יכולת ביטול הרשמת משתמש (soft delete)

#### טכני
- גישה מוגנת: בדיקת UID מול רשימת אדמינים ב-Firestore Security Rules / env config — לא role בבסיס הנתונים הרגיל
- כל פעולת כתיבה של אדמין נרשמת ב-audit log פשוט (מי עשה מה ומתי)
- הדשבורד לא חייב להיות mobile-first — עיצוב desktop מספיק

### מודול D: תקשורת, שיתוף והתראות

#### יצירת הודעת וואטסאפ
- כפתור "שתף לוואטסאפ" יוצר טקסט מוכן לשליחה, לדוגמה:
  ```
  היי! 🎉
  אנחנו מארגנים: מסיבת סוף שנה כיתה ה'
  בתאריך: שישי, 30.5
  
  כנסו לקישור הזה ורשמו מה אתם מביאים:
  https://bringit.app/event/d3b07384-d113-4ec6-a5d7
  ```
- שימוש ב-`whatsapp://send?text=...` למובייל

#### חזרת אורח לאירוע חסום
- אורח שפותח קישור של אירוע במצב **חסום** רואה מסך ייעודי:
  - הודעה: "האירוע הזה נחסם על ידי המארגן"
  - כפתור: "שלח שאלה למארגן"
- לחיצה על הכפתור פותחת **טופס פניה קצר** (שם, הודעה חופשית עד 300 תווים) שנשלח למארגן **באימייל** (כתובת האימייל של המארגן שמורה בחשבון Google שלו)
  - הנמען: המארגן. נושא: `[BringIt] פניה בנוגע לאירוע: {שם האירוע}`
  - הגנה על פרטיות: כתובת המייל של המארגן אינה חשופה לאורח — השליחה דרך השרת
  - חלופה: אם למארגן יש מספר וואטסאפ שמור בפרופיל — מוצג גם כפתור "פנה בוואטסאפ" (deep link)
- אירוע **מוקפא** (frozen) שונה מחסום: האורח רואה את הרשימה במצב קריאה, ללא אפשרות פניה
- אירוע **מחוק** מחזיר שגיאת 404 רגילה

#### התראת סיום אירוע למארגן
- לאחר מעבר תאריך האירוע, המארגן מקבל התראה באפליקציה (ובאימייל אם רשום)
- ההתראה כוללת סיכום: מספר משתתפים, % כיסוי רשימה, רשימת פריטים שלא נלקחו
- בהתראה — שלושה כפתורי פעולה ישירים: **מחק / הקפא / חסום**
- אם המארגן לא פעל תוך 7 ימים — האירוע עובר אוטומטית למצב הקפאה

#### סיכום סטטוס (למשתמשים רשומים)
- התראה / בנר: "יש לך 2 אירועים השבת. עליך להביא: עוגה ×1, מיץ ×2"
- לא בשלב הראשון — ניתן לממש כאיצ'אפ עתידי

## ארכיטקטורה טכנית (המלצות)

### Frontend
- Framework: React (Next.js) או Svelte — Mobile-First
- Styling: Tailwind CSS
- State: Zustand / Jotai לניהול מצב מקומי

### Backend & Database
- **Firebase Firestore** (מומלץ) — סנכרון בזמן אמת, Auth מובנה, אחסון LocalStorage-like עם Security Rules

### Auth
- Google OAuth2 (Firebase Auth / NextAuth)
- אנונימי: ללא אימות — UUID כטוקן גישה

### Hosting
- Vercel (Next.js) / Firebase Hosting

## מבנה מסד הנתונים (סכמה לוגית)

```
events/
  {eventId}/
    name: string
    date: timestamp
    description: string
    mode: "prescribed" | "open"
    status: "active" | "frozen" | "blocked" | "deleted"
    allowUnclaim: boolean         // האם אורחים יכולים לבטל בחירה
    visibilityMode: "open" | "private"  // מי רואה את שמות הבוחרים
    clonedFromEventId: string | null  // מאיזה אירוע שוכפל
    createdBy: userId              // חובה — יצירת אירוע מחייבת הרשמה
    createdAt: timestamp
    shareToken: string (= eventId)

    items/
      {itemId}/
        name: string
        quantity: number          // רק ב-prescribed
        claimedBy: [{ userId, nickname, amount }]
        addedBy: userId | null    // רק ב-open

users/
  {userId}/
    displayName: string
    email: string
    createdAt: timestamp
    eventsOwned: [eventId]
    eventsAttending: [{ eventId, itemId, amount }]
```

## החלטות שהתקבלו

| נושא | החלטה |
|------|-------|
| ביטול בחירה ע"י אורח | הגדרה בינארית ברמת האירוע — המארגן בוחר אם מותר או לא |
| ביטול בחירה ע"י מארגן | תמיד מותר, ללא תלות בהגדרה |
| מחיקה אוטומטית | לא — אין מחיקה אוטומטית; המארגן מחליט |
| פעולה אחרי תאריך האירוע | התראה למארגן עם 3 אפשרויות: מחק / הקפא / חסום |
| ברירת מחדל ללא פעולה | הקפאה אחרי 7 ימים של אי-פעולה |
| שימוש חוזר ברשימה | שכפול אירוע — כל הפריטים מועתקים (כולל כאלה שאורחים הוסיפו ב-open list), כמויות מועתקות, כל הבחירות מתאפסות — התחלה ריקה |
| נראות בחירות | ברירת מחדל: פתוח (כולם רואים מי בחר מה); אפשרות פרטי שמסתיר שמות מאורחים. המארגן רואה הכל תמיד |
| אורח חוזר לאירוע חסום | מסך ייעודי עם הודעה + טופס פניה למארגן באימייל (דרך השרת); חלופת וואטסאפ אם המארגן שמר מספר |
| הבדל חסום מול מוקפא | חסום: אין גישה + אפשרות פניה. מוקפא: קריאה בלבד, ללא פניה |
| מי יכול ליצור אירוע | רק משתמשים רשומים עם Google — אין יצירת אירוע אנונימית |
